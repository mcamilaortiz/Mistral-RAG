<h2>TrainingJob operator<a name="trainingjob-operator"></a></h2>

<p>Training job operators reconcile your specified training job spec to SageMaker by launching it for you in SageMaker. You can learn more about SageMaker training jobs in the SageMaker <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_CreateTrainingJob.html">CreateTrainingJob API documentation</a>. </p>

<p><strong>Topics</strong>
+ <a href="#create-a-trainingjob-using-a-simple-yaml-file">Create a TrainingJob Using a YAML File</a>
+ <a href="#create-a-trainingjob-using-a-helm-chart">Create a TrainingJob Using a Helm Chart</a>
+ <a href="#list-training-jobs">List TrainingJobs</a>
+ <a href="#describe-a-training-job">Describe a TrainingJob</a>
+ <a href="#view-logs-from-training-jobs">View Logs from TrainingJobs</a>
+ <a href="#delete-training-jobs">Delete TrainingJobs</a></p>

<h3>Create a TrainingJob Using a YAML File<a name="create-a-trainingjob-using-a-simple-yaml-file"></a></h3>

<ol>
<li><p>Download the sample YAML file for training using the following command: </p>

<p><code>
wget https://raw.githubusercontent.com/aws/amazon-sagemaker-operator-for-k8s/master/samples/xgboost-mnist-trainingjob.yaml
</code></p></li>
<li><p>Edit the <code>xgboost-mnist-trainingjob.yaml</code> file to replace the <code>roleArn</code> parameter with your <code>&lt;sagemaker-execution-role&gt;</code>, and <code>outputPath</code> with your Amazon S3 bucket that the SageMaker execution role has write access to. The <code>roleArn</code> must have permissions so that SageMaker can access Amazon S3, Amazon CloudWatch, and other services on your behalf. For more information on creating an SageMaker ExecutionRole, see <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/sagemaker-roles.html#sagemaker-roles-createtrainingjob-perms">SageMaker Roles</a>. Apply the YAML file using the following command: </p>

<p><code>
kubectl apply -f xgboost-mnist-trainingjob.yaml
</code></p></li>
</ol>

<h3>Create a TrainingJob Using a Helm Chart<a name="create-a-trainingjob-using-a-helm-chart"></a></h3>

<p>You can use Helm Charts to run TrainingJobs. </p>

<ol>
<li><p>Clone the GitHub repository to get the source using the following command: </p>

<p><code>
git clone https://github.com/aws/amazon-sagemaker-operator-for-k8s.git
</code></p></li>
<li><p>Navigate to the <code>amazon-sagemaker-operator-for-k8s/hack/charts/training-jobs/</code> folder and edit the <code>values.yaml</code> file to replace values like <code>rolearn</code> and <code>outputpath</code> with values that correspond to your account. The RoleARN must have permissions so that SageMaker can access Amazon S3, Amazon CloudWatch, and other services on your behalf. For more information on creating an SageMaker ExecutionRole, see <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/sagemaker-roles.html#sagemaker-roles-createtrainingjob-perms">SageMaker Roles</a>. </p></li>
</ol>

<h4>Create the TrainingJob<a name="create-the-training-job"></a></h4>

<p>With the roles and Amazon S3 buckets replaced with appropriate values in <code>values.yaml</code>, you can create a training job using the following command: </p>

<p><code>
helm install . --generate-name
</code></p>

<p>Your output should look like the following: </p>

<p><code>
NAME: chart-12345678
LAST DEPLOYED: Wed Nov 20 23:35:49 2019
NAMESPACE: default
STATUS: deployed
REVISION: 1
TEST SUITE: None
NOTES:
Thanks for installing the sagemaker-k8s-trainingjob.
</code></p>

<h4>Verify Your Training Helm Chart<a name="verify-your-training-helm-chart"></a></h4>

<p>To verify that the Helm Chart was created successfully, run: </p>

<p><code>
helm ls
</code></p>

<p>Your output should look like the following: </p>

<p><code>
NAME                    NAMESPACE       REVISION        UPDATED                                 STATUS          CHART                           APP VERSION
chart-12345678        default         1               2019-11-20 23:35:49.9136092 +0000 UTC   deployed        sagemaker-k8s-trainingjob-0.1.0
rolebased-12345678    default         1               2019-11-20 23:14:59.6777082 +0000 UTC   deployed        sagemaker-k8s-operator-0.1.0
</code></p>

<p><code>helm install</code> creates a <code>TrainingJob</code> Kubernetes resource. The operator launches the actual training job in SageMaker and updates the <code>TrainingJob</code> Kubernetes resource to reflect the status of the job in SageMaker. You incur charges for SageMaker resources used during the duration of your job. You do not incur any charges once your job completes or stops. </p>

<p><strong>Note</strong>: SageMaker does not allow you to update a running training job. You cannot edit any parameter and re-apply the file/config. Either change the metadata name or delete the existing job and create a new one. Similar to existing training job operators like TFJob in Kubeflow, <code>update</code> is not supported. </p>

<h3>List TrainingJobs<a name="list-training-jobs"></a></h3>

<p>Use the following command to list all jobs created using the Kubernetes operator: </p>

<p><code>
kubectl get TrainingJob
</code></p>

<p>The output listing all jobs should look like the following: </p>

<p><code>
kubectl get trainingjobs
NAME                        STATUS       SECONDARY-STATUS   CREATION-TIME          SAGEMAKER-JOB-NAME
xgboost-mnist-from-for-s3   InProgress   Starting           2019-11-20T23:42:35Z   xgboost-mnist-from-for-s3-examplef11eab94e0ed4671d5a8f
</code></p>

<p>A training job continues to be listed after the job has completed or failed. You can remove a <code>TrainingJob</code> job from the list by following the Delete a Training Job steps. Jobs that have completed or stopped do not incur any charges for SageMaker resources. </p>

<h4>TrainingJob Status Values<a name="training-job-status-values"></a></h4>

<p>The <code>STATUS</code> field can be one of the following values: 
+ <code>Completed</code> 
+ <code>InProgress</code> 
+ <code>Failed</code> 
+ <code>Stopped</code> 
+ <code>Stopping</code> </p>

<p>These statuses come directly from the SageMaker official <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_DescribeTrainingJob.html#SageMaker-DescribeTrainingJob-response-TrainingJobStatus">API documentation</a>. </p>

<p>In addition to the official SageMaker status, it is possible for <code>STATUS</code> to be <code>SynchronizingK8sJobWithSageMaker</code>. This means that the operator has not yet processed the job. </p>

<h4>Secondary Status Values<a name="secondary-status-values"></a></h4>

<p>The secondary statuses come directly from the SageMaker official <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_DescribeTrainingJob.html#SageMaker-DescribeTrainingJob-response-SecondaryStatus">API documentation</a>. They contain more granular information about the status of the job. </p>

<h3>Delete TrainingJobs<a name="delete-training-jobs"></a></h3>

<p>Use the following command to stop a training job on Amazon SageMaker: </p>

<p><code>
kubectl delete trainingjob xgboost-mnist-from-for-s3
</code></p>

<p>This command removes the SageMaker training job from Kubernetes. This command returns the following output: </p>

<p><code>
trainingjob.sagemaker.aws.amazon.com "xgboost-mnist-from-for-s3" deleted
</code></p>

<p>If the job is still in progress on SageMaker, the job stops. You do not incur any charges for SageMaker resources after your job stops or completes. </p>

<p><strong>Note</strong>: SageMaker does not delete training jobs. Stopped jobs continue to show on the SageMaker console. The <code>delete</code> command takes about 2 minutes to clean up the resources from SageMaker. </p>

<h4>Hyperparameter Tuning Job Status Values<a name="hyperparameter-tuning-job-status-values"></a></h4>

+ <code>InProgress</code> 
+ <code>Failed</code> 
+ <code>Stopped</code> 
+ <code>Stopping</code> </p>

For more information about how these are determined, see <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_TrainingJobStatusCounters.html">TrainingJobStatusCounters</a> in the SageMaker API documentation. </p>

<h4>Best TrainingJob<a name="best-training-job"></a></h4>

<p>This column contains the name of the <code>TrainingJob</code> that best optimized the selected metric. </p>

<p><code>
kubectl describe hyperparametertuningjob xgboost-mnist-hpo
</code></p>

<p>To see detailed information about the <code>TrainingJob</code>, run: </p>

<p><code>
kubectl describe trainingjobs &lt;job name&gt;
</code></p>

<h2>HostingDeployment operator<a name="hosting-deployment-operator"></a></h2>

<p>HostingDeployment operators support creating and deleting an endpoint, as well as updating an existing endpoint, for real-time inference. You can learn more about SageMaker inference in the SageMaker <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_CreateEndpoint.html">CreateEndpoint API documentation</a>. </p>

<h3>Configure a HostingDeployment Resource<a name="configure-a-hostingdeployment-resource"></a></h3>

<p>Download the sample YAML file for the hosting deployment job using the following command: </p>

<p><code>
wget https://raw.githubusercontent.com/aws/amazon-sagemaker-operator-for-k8s/master/samples/xgboost-mnist-hostingdeployment.yaml
</code></p>

<p>The <code>xgboost-mnist-hostingdeployment.yaml</code> file has the following components that can be edited as required: 
+ <em>ProductionVariants</em>. A production variant is a set of instances serving a single model. SageMaker load-balances between all production variants according to set weights. 
+ <em>Models</em>. A model is the containers and execution role ARN necessary to serve a model. It requires at least a single container. 
+ <em>Containers</em>. A container specifies the dataset and serving image. If you are using your own custom algorithm instead of an algorithm provided by SageMaker, the inference code must meet SageMaker requirements. For more information, see <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/your-algorithms.html">Using Your Own Algorithms with SageMaker</a>. </p>

<h3>Create a HostingDeployment<a name="create-a-hostingdeployment"></a></h3>

</code></p>

<p>SageMaker creates an endpoint with the specified configuration. You incur charges for SageMaker resources used during the lifetime of your endpoint. You do not incur any charges once your endpoint is deleted. </p>

<h4>HostingDeployment Status Values<a name="hostingdeployment-status-values"></a></h4>

<p>The status field can be one of several values: 
+ <code>SynchronizingK8sJobWithSageMaker</code>: The operator is preparing to create the endpoint. 
+ <code>ReconcilingEndpoint</code>: The operator is creating, updating, or deleting endpoint resources. If the HostingDeployment remains in this state, use <code>kubectl describe</code> to see the reason in the <code>Additional</code> field. 
+ <code>OutOfService</code>: Endpoint is not available to take incoming requests. 
+ <code>Creating</code>: <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_CreateEndpoint.html">CreateEndpoint</a> is executing. 
+ <code>Updating</code>: <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_UpdateEndpoint.html">UpdateEndpoint</a> or <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_UpdateEndpointWeightsAndCapacities.html">UpdateEndpointWeightsAndCapacities</a> is executing. 
+ <code>SystemUpdating</code>: Endpoint is undergoing maintenance and cannot be updated or deleted or re-scaled until it has completed. This maintenance operation does not change any customer-specified values such as VPC config, KMS encryption, model, instance type, or instance count. 
+ <code>RollingBack</code>: Endpoint fails to scale up or down or change its variant weight and is in the process of rolling back to its previous configuration. Once the rollback completes, endpoint returns to an <code>InService</code> status. This transitional status only applies to an endpoint that has autoscaling enabled and is undergoing variant weight or capacity changes as part of an <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_UpdateEndpointWeightsAndCapacities.html">UpdateEndpointWeightsAndCapacities</a> call or when the <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_UpdateEndpointWeightsAndCapacities.html">UpdateEndpointWeightsAndCapacities</a> operation is called explicitly. 
+ <code>InService</code>: Endpoint is available to process incoming requests. 
+ <code>Deleting</code>: <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_DeleteEndpoint.html">DeleteEndpoint</a> is executing. 
+ <code>Failed</code>: Endpoint could not be created, updated, or re-scaled. Use <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_DescribeEndpoint.html#SageMaker-DescribeEndpoint-response-FailureReason">DescribeEndpoint:FailureReason</a> for information about the failure. <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/API_DeleteEndpoint.html">DeleteEndpoint</a> is the only operation that can be performed on a failed endpoint. </p>

<h3>Invoking the Endpoint<a name="invoking-the-endpoint"></a></h3>

<p>Once the endpoint status is <code>InService</code>, you can invoke the endpoint in two ways: using the AWS CLI, which does authentication and URL request signing, or using an HTTP client like cURL. If you use your own client, you need to do AWS v4 URL signing and authentication on your own. </p>

<p>To invoke the endpoint using the AWS CLI, run the following command. Make sure to replace the Region and endpoint-name with your endpoint's Region and SageMaker endpoint name. This information can be obtained from the output of <code>kubectl describe</code>. </p>

<p>```</p>

<h1>Invoke the endpoint with mock input data.</h1>

<p>aws sagemaker-runtime invoke-endpoint \
  --region us-east-2 \
  --endpoint-name <endpoint name> \
  --body $(seq 784 | xargs echo | sed 's/ /,/g') \</p>

<blockquote>
  <p>(cat) \
    --content-type text/csv &gt; /dev/null
  ```</p>
</blockquote>

<p>For example, if your Region is <code>us-east-2</code> and your endpoint config name is <code>host-xgboost-f56b6b280d7511ea824b129926example</code>, then the following command would invoke the endpoint: </p>

<p>```
aws sagemaker-runtime invoke-endpoint \
  --region us-east-2 \
  --endpoint-name host-xgboost-f56b6b280d7511ea824b1299example \
  --body $(seq 784 | xargs echo | sed 's/ /,/g') \</p>

<blockquote>
  <p>(cat) \
    --content-type text/csv &gt; /dev/null
  4.95847082138
  ```</p>
</blockquote>

<p>Here, <code>4.95847082138</code> is the prediction from the model for the mock data. </p>

<h3>Update HostingDeployment<a name="update-hostingdeployment"></a></h3>

<ol>
<li><p>Once a HostingDeployment has a status of <code>InService</code>, it can be updated. It might take about 10 minutes for HostingDeployment to be in service. To verify that the status is <code>InService</code>, use the following command: </p>

<p><code>
kubectl get hostingdeployments
</code></p></li>
<li><p>The HostingDeployment can be updated before the status is <code>InService</code>. The operator waits until the SageMaker endpoint is <code>InService</code> before applying the update. </p>

<p>To apply an update, modify the <code>hosting.yaml</code> file. For example, change the <code>initialInstanceCount</code> field from 1 to 2 as follows: </p>

<p><code>
apiVersion: sagemaker.aws.amazon.com/v1
kind: HostingDeployment
metadata:
  name: host-xgboost
spec:
    region: us-east-2
    productionVariants:
        - variantName: all-traffic
          modelName: xgboost-model
          initialInstanceCount: 2
          instanceType: ml.c5.large
    models:
        - name: xgboost-model
          executionRoleArn: arn:aws:iam::123456789012:role/service-role/AmazonSageMaker-ExecutionRole
          primaryContainer: xgboost
          containers:
            - xgboost
    containers:
        - containerHostname: xgboost
          modelDataUrl: s3://my-bucket/inference/xgboost-mnist/model.tar.gz
          image: 123456789012.dkr.ecr.us-east-2.amazonaws.com/xgboost:latest
</code></p></li>
<li><p>Save the file, then use <code>kubectl</code> to apply your update as follows. You should see the status change from <code>InService</code> to <code>ReconcilingEndpoint</code>, then <code>Updating</code>. </p>

<p>```
$ kubectl apply -f hosting.yaml
hostingdeployment.sagemaker.aws.amazon.com/host-xgboost configured</p>

<p>$ kubectl get hostingdeployments
NAME           STATUS                SAGEMAKER-ENDPOINT-NAME
host-xgboost   ReconcilingEndpoint   host-xgboost-def0e83e0d5f11eaaa450a350abcdef</p>

<p>$ kubectl get hostingdeployments
NAME           STATUS     SAGEMAKER-ENDPOINT-NAME
host-xgboost   Updating   host-xgboost-def0e83e0d5f11eaaa450a3507abcdef
```</p></li>
</ol>

<p>SageMaker deploys a new set of instances with your models, switches traffic to use the new instances, and drains the old instances. As soon as this process begins, the status becomes <code>Updating</code>. After the update is complete, your endpoint becomes <code>InService</code>. This process takes approximately 10 minutes. </p>

<h3>Delete the HostingDeployment<a name="delete-the-hostingdeployment"></a></h3>

<ol>
<li><p>Use <code>kubectl</code> to delete a HostingDeployment with the following command: </p>

<p><code>
kubectl delete hostingdeployments host-xgboost
</code></p>

<p>Your output should look like the following: </p>

<p><code>
hostingdeployment.sagemaker.aws.amazon.com "host-xgboost" deleted
</code></p></li>
<li><p>To verify that the hosting deployment has been deleted, use the following command: </p>

<p><code>
kubectl get hostingdeployments
No resources found.
</code></p></li>
</ol>

<p>Endpoints that have been deleted do not incur any charges for SageMaker resources. </p>

<h2>HostingAutoscalingPolicy (HAP) Operator<a name="kubernetes-hap-operator"></a></h2>

<p>The HostingAutoscalingPolicy (HAP) operator takes a list of resource IDs as input and applies the same policy to each of them. Each resource ID is a combination of an endpoint name and a variant name. The HAP operator performs two steps: it registers the resource IDs and then applies the scaling policy to each resource ID. <code>Delete</code> undoes both actions. You can apply the HAP to an existing SageMaker endpoint or you can create a new SageMaker endpoint using the <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/hosting-deployment-operator.html#create-a-hostingdeployment">HostingDeployment operator</a>. You can read more about SageMaker autoscaling in the <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/endpoint-auto-scaling.html"> Application Autoscaling Policy documentation</a>.</p>

<p><strong>Note</strong><br />
In your <code>kubectl</code> commands, you can use the short form, <code>hap</code>, in place of <code>hostingautoscalingpolicy</code>.</p>

<p><strong>Topics</strong>
+ <a href="#kubernetes-hap-job-yaml">Create a HostingAutoscalingPolicy Using a YAML File</a>
+ <a href="#kubernetes-hap-list">List HostingAutoscalingPolicies</a>
+ <a href="#kubernetes-hap-describe">Describe a HostingAutoscalingPolicy</a>
+ <a href="#kubernetes-hap-update">Update a HostingAutoscalingPolicy</a>
+ <a href="#kubernetes-hap-delete">Delete a HostingAutoscalingPolicy</a>
+ <a href="#kubernetes-hap-update-delete-endpoint">Update or Delete an Endpoint with a HostingAutoscalingPolicy</a></p>

<h3>Create a HostingAutoscalingPolicy Using a YAML File<a name="kubernetes-hap-job-yaml"></a></h3>

<p>Use a YAML file to create a HostingAutoscalingPolicy (HAP) that applies a predefined or custom metric to one or multiple SageMaker endpoints.</p>

<p>Amazon SageMaker requires specific values in order to apply autoscaling to your variant. If these values are not specified in the YAML spec, the HAP operator applies the following default values.</p>

<p>```</p>

<h1>Do not change</h1>

<p>Namespace                    = "sagemaker"</p>

<h1>Do not change</h1>

<p>ScalableDimension            = "sagemaker:variant:DesiredInstanceCount"</p>

<h1>Only one supported</h1>

<p>PolicyType                   = "TargetTrackingScaling"</p>

<h1>This is the default policy name but can be changed to apply a custom policy</h1>

<p>DefaultAutoscalingPolicyName = "SageMakerEndpointInvocationScalingPolicy"
```</p>

<h4>Considerations for HostingAutoscalingPolicies for Multiple Endpoints and Variants<a name="kubernetes-hap-scaling-considerations"></a></h4>

<p>The following considerations apply when you use multiple resource IDs:
+ If you apply a single policy across multiple resource IDs, one PolicyARN is created per resource ID. Five endpoints have five PolicyARNs. When you run the <code>describe</code> command on the policy, the responses show up as one job and include a single job status.
+ If you apply a custom metric to multiple resource IDs, the same dimension or value is used for all the resource ID (variant) values. For example, if you apply a customer metric for instances 1-5, and the endpoint variant dimension is mapped to variant 1, when variant 1 exceeds the metrics, all endpoints are scaled up or down.
+ The HAP operator supports updating the list of resource IDs. If you modify, add, or delete resource IDs to the spec, the autoscaling policy is removed from the previous list of variants and applied to the newly specified resource ID combinations. Use the <a href="https://docs.aws.amazon.com/sagemaker/latest/dg/kubernetes-hap-operator.html#kubernetes-hap-describe">https://docs.aws.amazon.com/sagemaker/latest/dg/kubernetes-hap-operator.html#kubernetes-hap-describe</a> command to list the resource IDs to which the policy is currently applied.</p>

<h3>Update or Delete an Endpoint with a HostingAutoscalingPolicy<a name="kubernetes-hap-update-delete-endpoint"></a></h3>

<p>To update an endpoint that has a HostingAutoscalingPolicy (HAP), use the <code>kubectl</code> <code>delete</code> command to remove the HAP, update the endpoint, and then reapply the HAP.</p>

<p>To delete an endpoint that has a HAP, use the <code>kubectl</code> <code>delete</code> command to remove the HAP before you delete the endpoint.</p>
